# /plan 명령어

코딩 전에 포괄적인 구현 계획을 수립합니다.

## 사용법
```
/plan <기능 또는 작업 설명>
```

## 이 명령어의 역할

1. **호출**: `planner` 에이전트
2. **분석**: 요구사항과 코드베이스
3. **생성**: 단계별 구현 계획

## 계획 수립 프로세스

### 단계 1: 요구사항 분석
- 무엇을 구축해야 하는가?
- 성공 기준은 무엇인가?
- 제약사항은 무엇인가?

### 단계 2: 아키텍처 검토
- 기존 코드 구조 분석
- 영향받는 파일 식별
- 의존성 매핑

### 단계 3: 위험 평가
- 잠재적 문제 식별
- 위험 수준 할당 (높음/중간/낮음)
- 완화책 계획

### 단계 4: 단계별 계획
- 테스트 가능한 단위로 분할
- 구현 순서 정의
- 검증 단계 포함

## 계획 문서 구조

```markdown
## 개요
구현할 내용에 대한 간략한 설명

## 요구사항
- [ ] 요구사항 1
- [ ] 요구사항 2
- [ ] 요구사항 3

## 수정할 파일
- `src/module.py` - 새 함수 추가
- `tests/test_module.py` - 테스트 추가

## 새 파일
- `src/new_feature.py` - 기능 구현

## 구현 단계

### 단계 1: 테스트 파일 생성 [위험도: 낮음]
- `tests/test_feature.py` 생성
- 실패하는 테스트 케이스 추가
- 검증: `make test` 예상대로 실패

### 단계 2: 핵심 로직 구현 [위험도: 중간]
- `src/feature.py` 생성
- 메인 함수 구현
- 검증: 테스트 통과

### 단계 3: 오류 처리 추가 [위험도: 낮음]
- 검증 추가
- 커스텀 예외 추가
- 검증: 엣지 케이스 테스트 통과

## 테스트 전략
- 단위 테스트: 80% 커버리지 최소
- 통합 테스트: 주요 워크플로우
- 엣지 케이스: 잘못된 입력, 오류

## 위험 완화
| 위험 | 완화 |
|------|------|
| 기존 코드 깨짐 | 테스트 먼저 추가 |
| 성능 영향 | 구현 후 프로파일링 |

## 성공 기준
- [ ] 모든 테스트 통과
- [ ] Ruff 린트 통과
- [ ] Ty 타입체크 통과
- [ ] 커버리지 >= 80%
```

## 중요 규칙

1. **승인 전 코드 없음**
   - 플래너는 구현 코드를 작성하지 않음
   - 명시적 사용자 확인 대기

2. **수정 옵션**
   - "수정: [변경사항]" - 계획 조정
   - "대안: [접근법]" - 다른 전략

3. **승인**
   - "진행" 또는 "승인" - 구현 시작
   - 그 후 `/tdd`로 테스트 주도 개발

## 예제 세션

```
사용자: /plan JWT를 사용한 사용자 인증 추가

Claude:
## 구현 계획: JWT 인증

### 개요
API에 JWT 기반 인증 추가...

### 요구사항
- [ ] 사용자 로그인 엔드포인트
- [ ] 토큰 생성
- [ ] 토큰 검증 미들웨어
...

[전체 계획 문서]

---
다음 중 선택하세요:
- 계획 수정 ("수정: [변경사항]")
- 대안 시도 ("대안: [접근법]")
- 구현 진행 ("진행")
```

## 계획 수립 후

1. 사용자가 계획 승인
2. `/tdd` - TDD로 구현
3. `/code-review` - 변경사항 리뷰
4. 변경사항 커밋

## 관련 명령어

- `/tdd` - 테스트 주도 개발 시작
- `/code-review` - 구현 리뷰
- `/refactor-clean` - 코드 정리
